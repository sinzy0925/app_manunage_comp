# YouTube字幕要約を「まるなげ vs スキル vs アプリ」で比較した——時間・精度・コストの検証記録

> **ステータス:** 計測完了後に結果セクションを埋めて公開する想定の下書きです。

## はじめに

SNSでは「AI社員」「仕事をAIにまるなげ」といった投稿をよく見かけます。仕事が楽になるのは理解できますが、**本当に得なのか**は投稿からは見えにくい、と感じていました。

業務効率化なら、本来は次の関係を見るはずです。

```
削減した時間 × 自分の単価  vs  その時間を作るために AI に払う単価
```

AI側のコストが小さければ得。逆なら、楽でも割に合わない可能性があります。

本記事では、この疑問を数字で確認するために、**1つの YouTube URL から字幕を取得して要約する**という同じ作業を、3つのやり方で比較した記録をまとめます。

| 比較軸 | 内容 |
|---|---|
| やり方 | まるなげ / スキル / アプリ |
| モデル | Composer2.5（Fastでない方） / Claude-Opus-4.8-low-Thinking |
| 指標 | 時間 / 要約の精度 / トークンコスト（USD） |
| 対象URL | https://www.youtube.com/watch?v=koBLOf-53_g |
| 試行回数 | 方式 × モデルごとに各3回（計18回） |

使用ツールは **Cursor** のみです。

---

## 検証前の想定（仮説）

先に想定だけ書いておきます。結果は計測後にそのまま公表し、想定と違えばそちらを正とします。

**本線の想定:**

- 同じ仕事なら、高単価モデルでなくても **Cursor + Composer2.5** で十分戦えるはず
- 「高いモデル必須」という空気は過大評価では、という見立て

**別枠の想定（本線とは切り分け）:**

- **Gemini API の無料枠**でも、文章要約なら十分戦える可能性がある
- 今回は YouTube が対象で学習されてもよい前提のため、別途無料枠も試す
- コストだけ見ると、アプリに Gemini API を組み込んで無料枠で回すのが強そう、という見立て

---

## なぜ比較の前に「土俵づくり」をしたか

最初から3方式を並べたわけではありません。次の順でテンプレートを作りながら揃えました。

1. まるなげプロンプトを実行
2. そのリポジトリでスキル作成を指示
3. スキルのみ新規リポジトリへ入れてスキル実行
4. そのリポジトリでアプリ作成を指示
5. アプリとプロンプトを実行

要約結果を並べたところ、**アプリ版が一番弱かった**ため、数回改善しました。すると品質は上がりましたが、**他方式より強くなりすぎた**状態になりました。

アプリ化はバグ修正・パイプライン調整・プロンプト整備がつきもので、完成度の差が大きく出やすい。比較として歪むため、次を実施しました。

- アプリのリポジトリで、処理全体を回せる「まるなげ用プロンプト」を作成・実行
- そのまるなげプロンプトからスキルを再作成

その結果、**アプリ / まるなげ / スキルの要約レベル感がだいたい同じ**ところまで揃い、本番の計測に入りました。

> 本記事の比較は「ありのままの実力差」ではなく、**同程度の要約品質まで揃えたうえでの運用差**（時間・手間・コスト）を見る設計です。

## 比較ルール

### 3つのやり方

| 方式 | 概要 |
|---|---|
| **まるなげ** | プロンプトに YouTube URL を渡し、字幕取得〜要約までをエージェントに任せる |
| **スキル** | Cursor Skill（`youtube-ja-summary`）の手順とスクリプトに従って実行 |
| **アプリ** | 字幕取得・クリーニング・バンドル生成は手動パイプライン、要約は IDE エージェント |

### モデル

- **Composer2.5**（Fast でない方）—— Cursor で比較的安いモデル
- **Claude-Opus-4.8-low-Thinking** —— 高単価側

### 計測方法

| 指標 | 測り方 |
|---|---|
| **時間** | プロンプト入力直前から結果出力後まで（アプリは前後のパイプライン処理時間も含む） |
| **精度** | 全試行終了後、要約本文のみを観点別に比較（後述） |
| **トークン・USD** | [Cursor Usage](https://cursor.com/dashboard/usage) の CSV を取得し、[モデル単価](https://cursor.com/ja/docs/models-and-pricing) で換算 |

### 再現性のための条件

- 対象 URL は固定（`koBLOf-53_g`）
- 各試行は**完全に新規フォルダ**で実施
- フォルダの初期内容は `比較予定/` 配下の各方式テンプレート
- 要約の精度が同程度になるよう、プロンプトは事前に編集済み
- 方式 × モデルごとに **3回** 実行（計18回）

---

## 各方式の実行手順

### まるなげ

新規フォルダ（`比較予定/まるなげ` 相当）を開き、まるなげ用プロンプトを実行します。YouTube URL のみを入力し、字幕取得から要約 MD 出力までを一連で行います。

### スキル

新規フォルダ（`比較予定/スキル` 相当）を開き、スキル `youtube-ja-summary` に従います。

```powershell
$env:PYTHONIOENCODING = "utf-8"
python .cursor/skills/youtube-ja-summary/scripts/run_pipeline.py "https://www.youtube.com/watch?v=koBLOf-53_g"
```

パイプライン成功後、生成された `segments/` と `internal_memo.json` をもとに要約 MD を作成します。

### アプリ

新規フォルダ（`比較予定/アプリ` 相当）を開き、次の3ステップです。

**Step 1: 字幕〜バンドル**

```powershell
python scripts/run_pipeline.py "https://www.youtube.com/watch?v=koBLOf-53_g" --steps fetch,clean,bundle
```

**Step 2: IDE で要約**

```
@youtube-jimaku-summary
jimaku/ の要約前テキストを元に要約して
```

**Step 3: レポート生成**

```powershell
python scripts/run_pipeline.py --video-id koBLOf-53_g --steps render
```

## 精度の評価方法

時間・コストとは別に、**要約本文だけ**を観点別に比較しました。総合ランキングより、どの要約がどの観点で強い／弱いかを見る設計です。

評価観点（各 0〜2点、満点10）:

| 観点 | 内容 |
|---|---|
| A. 事実忠実性 | 幻覚（字幕にない事実の追加）の少なさ |
| B. エンティティ一貫性 | 組織・人物・製品の混同の少なさ |
| C. 情報の充足 | 結論・章立て・セグメントの抜けの少なさ |
| D. 構造・読みやすさ | 実務で使える密度と流れ |
| E. 形式遵守 | 必須セクション・メタ文・セグメント数の整合 |

評価用プロンプトはリポジトリ内の `要約比較プロンプト.md` に記載しています。参照字幕（`cleaned.txt` 等）がある場合は一次ソースとして使用します。
