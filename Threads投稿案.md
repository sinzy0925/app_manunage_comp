# Threads投稿案（推奨順・穏やかな実況口調）

疑問 → 仮説 → 別枠（2.5）→ ルール → 前準備 → 日次実況 → 結論、の順。  
数字・所感は当日の内容に差し替える。

---

## 1. キッカケ（なぜ測るのか）

```text
最近、ずっと引っかかっていることがあります。

SNSでよく見る
「AI社員」「仕事をAIにまるなげ」。
正直、あまりピンとこなかった。

仕事は楽になるかもしれない。
でも、コストはどうなの？

普通、業務効率化ならコストも下げるはずです。
人間の作業時間が減ったのはわかる。

見るべきなのは、たぶんここです。

・削減した時間 × 自分の単価
・その時間を作るために AI に払う単価

AI側が小さければ得。
でも本当にそうなっているのかは、投稿からは見えにくい。

だから同じ土俵で同じ処理をさせて、測定する。
感想ではなく、時間と精度とトークンコストで比較する。

具体的には、1つのYouTube URLから字幕データを取得して要約する作業を、
まるなげ / スキル / アプリの3つのやり方で比較します。
```

## 2. 想定結論（仮説）

```text
先に、いまの想定は以下の通りです。

想定:
同じ仕事なら、高単価モデルでなくても
Cursor + Composer2.5 で十分戦えるはず。
高いモデル必須、みたいな空気は過大評価だと思っている。

ただ、これは予想です。
検証はきっちりやって、出た数字はそのまま出します。
想定と違う結果なら、そっちを正しい答えとして書きます。

先に勝ち宣言はしません。測ってから言います。
```

## 2.5. 別枠（Gemini API 無料枠）

本線（Cursor上のモデル比較）とは切り分けて書く。

```text
別枠で、もう一つ見ていることがあります。

Gemini API の無料枠です。
文章の要約なら、無料枠でも十分戦えるのでは、と思っています。

今回は YouTube が対象で、学習されてもよい前提なので、無料枠も試してみます。
コストだけ見ると、アプリに Gemini API を組み込んで無料枠で回すのが強そう、というのがいまの見立てです。

これは本線（Cursor + Composer2.5 vs 高単価モデル）とは別枠です。
本線の数字とは切り分けて書きます。
```

## 3. 何を比較するか（ルール）

```text
比較のルールは以下の通りです。

やり方は3つ。
まるなげ / スキル / アプリ

モデルは2つ。
Composer2.5（Fastでない方）
Claude-Opus-4.8-low-Thinking

見るのは3つ。
1. かかった時間
2. 要約の精度
3. トークン量（ドル換算）
単価の参照はここ:
https://cursor.com/ja/docs/models-and-pricing

名前        Composer2.5　---　Claude Opus 4.8
入力　　　　　　　　$0.5  ---　$5	
キャッシュ書き込み　-　   ---　$6.25		　
キャッシュ読み取り　$0.2　---　$0.5	
出力　　　　　　　　$2.5　---　$25

同じYouTube URLで固定。
方式×モデルごとに各3回。

AI社員が得か損か、ではなく、
同じ仕事を違う形でやったときに何が残るかを見ます。
```

## 4. 前準備の物語

```text
本番の前に、土台づくりの話。

比較したいのは
まるなげ / スキル / アプリ。

最初から綺麗に並んでいたわけではなくて、
この順でテンプレを作りながら揃えました。

流れはこんな感じです。

まず、まるなげプロンプトを実行。
そのリポジトリでスキル作成を指示。
できたスキルだけを新規リポジトリに入れて実行。
そのリポジトリでアプリ作成を指示。
アプリとプロンプトを実行。

ここまでやって、要約結果を並べてみました。

すると、アプリ版が一番弱かった。

そこでアプリを改善。
一度ではなく、数回ほど手を入れました。

すると結果は良くなった。
ただ、良くなりすぎた。

他の方式より明らかに強くなって、このままだと比較として歪む感じでした。

アプリ化は修正がつきもので、差が出やすい。
だから調整した、という話です。

次は、アプリのリポジトリで
処理全体を回せる「まるなげ用プロンプト」を書いてもらい、実行。

アプリとまるなげの要約が、いい感じに並びました。

そこから、まるなげプロンプトをもとにスキルも作成。

いまは、アプリ / まるなげ / スキルの要約が
だいたい同じレベル感まで揃っています。

ここまでが前準備。
ここからが本番です。
```

### 本編開始の一言（任意）

## 5. 毎日の実況（本番）

1本終わるごとに投稿。結論は急がない。

### 基本テンプレ

```text
【検証 Day○ / ○/18】

方式: まるなげ or スキル or アプリ
モデル: Composer2.5 / Opus
所要: ○分○秒

所感（1行）:
例「手順は重いが安心感がある」

結論はまだ出さない。数字を積む段階です。
```

### まるなげの日（任意の補足）

```text
今日はまるなげ条件。

URLを渡して進める形は楽です。
ただし、楽さと得は別なので、時間とコストも一緒に残します。

結果: ○分 / 所感: ○○
```

### スキルの日（任意の補足）

```text
今日はスキル条件。

手順が固定される分、再現性は上がりやすい。
それが時間やコストにどう出るか、を見ています。

結果: ○分 / 所感: ○○
```

### アプリの日（任意の補足）

```text
今日はアプリ条件。

人がパイプラインを回し、AIは要約側。
前後のアプリ処理時間も含めて測ります。

結果: 合計○分 / 所感: ○○
```

### 区切り（半分・終盤の任意）

```text
○/18まで来ました。

いま見えている傾向だけ先に共有します。
- まるなげは ____
- スキルは ____
- アプリは ____

最終の結論は、全本揃ってからです。
```

## 投稿スケジュール目安

| タイミング | 投稿 |
|---|---|
| 初日 | 1 → 2 → 2.5 → 3（1スレッドでも可） |
| 計測開始前 or 翌日 | 4 |
| 計測中 | 5を日次 |
| 全本終了後 | 6 |

### 初日を1スレッドにする例

1. キッカケ  
2. 想定結論  
2.5. 別枠（Gemini）  
3. 比較ルール  

読者の入口は「コストの違和感」、出口は「数字で確認」にそろえる。
